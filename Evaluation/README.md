# Evaluation Directory

This folder contains the output and metadata for the XAI evaluation pipeline. It is generated by the code in `src/evaluation.py` and captures the first evaluation phase implemented so far: **Efficiency** and **Simplicity**.

## Contents

- `metrics_long.csv`
  - Master long-format table of evaluation metric results.
  - Contains per-run and per-instance rows for all computed metrics.
- `run_ledger.csv`
  - Records each evaluation experiment as a single row with a JSON snapshot of the configuration and environment.
  - Keeps heavy metadata out of `metrics_long.csv` while preserving reproducibility.
- `logs/`
  - Contains evaluation logging output.
  - At the time of this snapshot, `efficiency_simplicity_log.txt` documents the run(s) that created the current output.

## Implementation Summary

The evaluation pipeline in `src/evaluation.py` is designed to be:

- reproducible
- dataset-size aware
- consistent across explainers (SHAP and LIME)
- extensible to future phases (`Fidelity`, `Robustness`, `Stability`)

The current phase intentionally computes only metrics that can be derived from already-generated explanation artifacts. This avoids doing extra explanation generation during evaluation and keeps the first phase efficient.

### Build Order and Design Rationale

The evaluation order is deliberately:

1. Efficiency
2. Simplicity
3. Fidelity
4. Robustness
5. Stability

This order matches a practical evaluation strategy in which runtime and explanation compactness are assessed first, and later phases reuse the same infrastructure for more expensive perturbation-based metrics. The pipeline is therefore built in phases rather than all at once.

### Shared Infrastructure

#### Experiment Ledger

- `create_experiment()` writes a new row to `run_ledger.csv`.
- It includes the experiment ID, configuration snapshot, Python and library versions, and timestamp.
- This keeps the evaluation environment traceable and separates stable metadata from metric rows.

#### Metrics Schema

`metrics_long.csv` uses a fixed schema with columns:

- `experiment_id`, `domain`, `dataset`, `model`, `explainer`
- `metric_property`, `metric_name`, `instance_id`
- `repeat_idx`, `perturbation_id`, `repeat_seed`
- `deterministic`, `baseline_type`, `mask_fraction`
- `value`, `runtime_ms`, `kernel_width`, `background_size`
- `num_features`, `num_instances`, `random_state`, `status`, `timestamp`

The schema is intentionally broader than the current metrics so that future Fidelity, Robustness, and Stability rows can reuse the same table without structural changes.

#### Metric Normalization

The implementation normalizes metrics to make them comparable across datasets of different dimensionality. This is important because the six datasets used by the current experiment range from a handful of original features to many one-hot expanded columns.

- **Efficiency**: uses `runtime_ms_per_instance_per_feature`.
- **Simplicity**: uses entropy normalized by `log2(n_original_features)` and percentage of original features needed for 80%/90% attribution mass.

These normalization choices are grounded in the interpretability literature: explainers should be compared on their computational cost per unit of input complexity, and explanation compactness should be assessed relative to the actual number of meaningful input factors rather than raw encoded columns [1,2,3,4].

### Feature-Group Mapping

A central piece of shared infrastructure is the feature-group map. It maps model input columns back to original features before simplicity metrics are computed.

- One-hot encoded categorical features are rolled back up to their original variable name.
- The mapping is built from the saved `OneHotEncoder` stored in preprocessing metadata, not by heuristics.
- This prevents categorical variables with many levels from appearing artificially more complex than numerical features.

This approach mirrors best practice in XAI evaluation: explanations should be evaluated over the original semantic feature set, not the expanded encoded representation [4,5]. It also ensures that simplicity and later masking-based fidelity metrics operate at the same granularity.

### LIME Condition Parsing

LIME explanations are stored as condition strings such as `Income <= 45000.00` or `Education_Bachelor's <= 0.00`. The pipeline parses those strings back to the actual model input column name using a longest-token match against the known feature order. This prevents incorrect matches like `Age` matching inside `AgeGroup` and ensures that LIME explanations are aggregated correctly.

Failed parse attempts are recorded with `status='failed'` instead of being silently dropped.

### Efficiency

`compute_efficiency()` reads runtime metadata already stored by the explanation generation step. It does not regenerate explanations.

Metrics recorded for each explainer run:

- `runtime_ms_per_instance`
- `runtime_ms_per_instance_per_feature`

Both `background_size` and `kernel_width` are logged as covariates for LIME, but not baked into the shared normalized efficiency metric. This is intentional because SHAP (`TreeExplainer` with `tree_path_dependent`) does not use a background dataset, so pooling those costs would distort cross-explainer comparisons.

### Simplicity

Simplicity is computed per instance after aggregating attributions to original features.

For SHAP:

- Absolute SHAP values are loaded from `shap_values.npy`.
- Values are aggregated to original features using the feature-group map.
- Absolute magnitudes are used because explanation simplicity is a function of attribution concentration, not signed effect.

For LIME:

- The pipeline reads `lime_explanations.csv`.
- Each LIME condition is mapped back to a model input column.
- Absolute weights are summed for each original feature group.
- Instances with no successfully parsed conditions are kept in the table with `status='failed'`.

Simplicity metrics:

- `normalized_entropy_complexity`
- `pct_features_for_80pct_mass`
- `pct_features_for_90pct_mass`

Entropy is normalized by `log2(n_original_features)` so that a maximally uniform attribution distribution maps to 1.0 regardless of dataset size; this makes the metric comparable across datasets with different original feature counts.

## How to Use

1. Create a new experiment with `src/evaluation.py:create_experiment()`.
2. Run `src/evaluation.py:run_efficiency_and_simplicity()` for each dataset/model combination.
3. Review the generated rows in `Evaluation/metrics_long.csv` and the associated log output in `Evaluation/logs/`.

## Notes for Future Extensions

- The current code is intentionally built so that `Fidelity`, `Robustness`, and `Stability` can reuse the same `metrics_long.csv` schema and the same feature-group mapping.
- Once those phases are added, rows can be appended without changing the existing output format.
- The `deterministic` and `status` fields are already available to support later analyses that differentiate deterministic explainers from randomized ones, and that keep failed parsing rows visible.
